## 1. 测试概述

### 1.1 测试目标

本次测试旨在全面验证 LEMS 实验设备管理系统的功能完整性、数据一致性和并发安全性，重点验证以下核心指标：系统功能是否满足需求规格说明书中的所有功能需求；设备库存扣减在并发场景下是否安全可靠（available_quantity 不会出现负数）；角色权限控制是否有效隔离（学生无法访问管理员接口）；各微服务之间的 Feign 调用是否正常且降级策略有效。

### 1.2 测试范围

- 功能测试：覆盖所有 API 接口（认证、用户、设备、借用、报表）。
- 并发测试：重点测试借用申请接口（`/borrow/apply`）在高并发下的库存安全性。
- 权限测试：验证三种角色（ADMIN、TEACHER、STUDENT）的接口访问权限。
- 集成测试：验证微服务间 Feign 调用、Redis 分布式锁、JWT 鉴权链路的完整性。

## 3. 并发压力测试（JMeter）

### 3.1 测试场景：借用申请接口并发安全性

这是本系统最关键的测试场景，验证设备库存扣减在并发条件下的安全性。

**测试目的**：模拟多个用户同时借用同一台设备（该设备仅剩 1 件可用），验证最终只有 1 个用户成功，且数据库 `available_quantity` 不会变为负数。

**测试环境准备**：

1. 创建测试设备：`total_quantity = 1, available_quantity = 1`
2. 确认 Redis 和所有微服务正常运行
3. 创建 10 个测试用户账号（user01 ~ user10）

**JMeter 测试计划配置**：

| 配置项 | 值 |
|--------|---|
| 测试线程数 | 10（模拟 10 个并发用户） |
| Ramp-Up 时间 | 1 秒（1 秒内启动所有线程） |
| 循环次数 | 1 |
| 目标 URL | POST `http://localhost:8080/api/v1/borrow/apply` |
| Content-Type | application/json |
| 请求体 | `{ "deviceId": 1, "borrowQuantity": 1, "startTime": "2025-05-01T09:00:00", "endTime": "2025-05-02T17:00:00", "purpose": "并发测试" }` |
| Authorization | 每个线程使用不同的用户 Token |

**JMeter 操作步骤**：

1. 打开 JMeter，创建测试计划"LEMS 并发借用测试"。
2. 添加"线程组"，设置线程数 10，Ramp-Up 1 秒，循环 1 次。
3. 添加"HTTP 请求"，配置 URL、方法、请求体。
4. 添加"HTTP Header Manager"，配置 `Authorization` 和 `Content-Type`。
5. 使用 CSV Data Set Config 文件读取 10 个用户的 Token。
6. 添加"查看结果树"和"汇总报告"监听器。
7. 点击"启动"执行测试。

### 3.2 测试结果

**测试执行记录**：

| 执行次数 | 并发数 | 设备可用库存 | 成功请求数 | 失败请求数 | 最终 available_quantity | 结果 |
|---------|--------|------------|-----------|-----------|----------------------|------|
| 第1次 | 10 | 1 | 1 | 9 | 0 | ✅ 通过 |
| 第2次 | 10 | 1 | 1 | 9 | 0 | ✅ 通过 |
| 第3次 | 10 | 1 | 1 | 9 | 0 | ✅ 通过 |
| 第4次 | 100 | 1 | 1 | 99 | 0 | ✅ 通过 |
| 第5次 | 100 | 5 | 5 | 95 | 0 | ✅ 通过 |

**详细分析（第 1 次执行）**：

- 10 个线程同时向 `/borrow/apply` 接口发送借用申请。
- 通过 Redis 分布式锁（`lock:device:1`），同一时刻只有一个请求能通过锁检查。
- 第一个获取锁的请求查询设备 `available_quantity = 1, version = N`。
- 执行乐观锁 SQL：`UPDATE device_info SET available_quantity = 0, version = N+1 WHERE id = 1 AND version = N AND available_quantity >= 1`，影响行数 = 1，扣减成功。
- 释放 Redis 锁后，后续请求获取锁并查询设备时，`available_quantity = 0`，触发"库存不足"异常。
- 最终结果：1 个成功，9 个失败（收到"库存不足"或"操作频繁"提示）。
- 数据库 `available_quantity = 0`，**未出现负数**。

**响应时间统计（100 并发场景）**：

| 指标 | 值 |
|------|---|
| 平均响应时间 | 45ms |
| 最小响应时间 | 12ms |
| 最大响应时间 | 230ms |
| 90% 响应时间 | 85ms |
| 错误率 | 0%（HTTP 层面），业务层面 99% 失败（预期行为） |
| 吞吐量 | 约 200 请求/秒 |

### 3.3 并发测试结论

- ✅ **库存安全性**：在所有并发测试场景中（10 并发、100 并发），设备 `available_quantity` 均未出现负数，乐观锁 + 分布式锁双重保障机制有效。
- ✅ **业务正确性**：当设备仅剩 1 件时，10 个并发请求中恰好只有 1 个成功，符合预期。
- ✅ **响应可接受**：平均响应时间在 50ms 以内，满足 500ms 的性能要求。
- ✅ **无异常错误**：未出现数据库异常、服务崩溃或数据不一致情况。
