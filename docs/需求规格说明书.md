## 1. 引言

### 1.1 编写目的

本文档是 LEMS 实验设备管理系统的需求规格说明书，旨在全面、准确地描述系统的功能需求与非功能需求。文档面向以下读者群体：项目开发团队成员（包括后端微服务开发工程师、前端 Vue 工程师）、软件测试工程师、项目评审老师及答辩委员会成员。通过阅读本文档，读者可以完整了解系统的业务背景、角色划分、核心功能流程、接口规范以及性能安全等方面的具体要求，为后续的架构设计、编码实现、测试验证和项目验收提供统一的依据和标准。

### 1.2 项目背景

随着高校实验室规模的持续扩大和教学科研活动的日益丰富，实验室所拥有的仪器设备种类和数量均呈现快速增长趋势。传统的设备管理方式主要依赖纸质台账和 Excel 表格，存在以下突出痛点：首先，设备借用预约流程需要师生线下填写纸质申请表，经多层审批后才能领取设备，整个流程耗时长、效率低，且容易出现信息丢失或记录不完整的情况；其次，设备库存状态无法实时更新，多个学生同时借用同一台设备时缺乏有效的并发控制机制，经常出现超借或库存数据不一致的问题；再者，设备维修报废等状态变更缺乏规范的流转记录，难以追踪设备的全生命周期历史；最后，缺乏数据统计与分析能力，管理人员无法直观了解设备的使用率和借用趋势，不利于科学决策和资源配置优化。

基于上述背景，本系统采用前后端分离架构，后端基于 Spring Cloud 微服务体系，前端基于 Vue 3 单页应用，旨在构建一个功能完善、性能可靠、安全可控的实验设备全生命周期管理平台，满足高校实验室在设备入库、借用预约、归还验收、维修报废、库存盘点与使用统计等方面的信息化管理需求。

## 2. 系统概述

### 2.1 系统目标

LEMS 系统的核心目标是实现高校实验室仪器设备的全生命周期数字化管理，具体包括以下几个方面：建立统一的设备台账数据库，实现设备信息的集中化、标准化管理；构建在线借用预约流程，替代传统的纸质申请方式，提高师生借用设备的便捷性和审批效率；实现库存数量的实时准确管理，通过乐观锁和分布式锁机制确保高并发场景下库存数据的一致性；提供设备使用情况的统计与分析功能，为实验室管理者的决策提供数据支撑；建立完整的设备维修报废记录，实现设备全生命周期的可追溯管理。

### 2.2 应用范围

本系统的应用范围涵盖高校内部的所有教学与科研实验室，包括物理实验室、化学实验室、计算机实验室、电子工程实验室等各类实验室场景。系统主要服务于三类用户角色：系统管理员（负责设备基础数据维护和用户管理）、教师或实验员（负责借用审批和实验室设备管理）、学生（通过系统借用设备并管理个人借用记录）。系统可部署在校园网内部服务器上，通过浏览器访问，支持桌面端使用。

## 3. 功能需求

### 3.1 角色定义

系统定义了三种核心用户角色，各角色具有不同的权限范围和操作能力：

**管理员（ADMIN）**：拥有系统的最高权限，负责全局管理操作。具体权限包括：管理和维护所有设备信息（新增、编辑、删除设备）、管理设备分类树、管理实验室信息、管理所有用户账号（启用/禁用）、查看所有借用记录和统计数据、配置数据字典。管理员是系统中唯一可以执行设备台账变更和用户管理的角色。

**教师/实验员（TEACHER）**：实验室的日常管理人员，拥有审批和管理权限。具体权限包括：审批学生提交的设备借用申请（批准或驳回）、查看和管理所负责实验室的成员列表、查看实验室设备的借用情况、提交设备维修申请。教师在借用审批流程中承担关键角色，确保设备借用的合理性和规范性。

**学生（STUDENT）**：系统的普通使用者，主要进行设备借用相关操作。具体权限包括：浏览和搜索可用设备列表、查看设备详细信息（名称、型号、可用数量、存放位置等）、提交设备借用申请、查看和管理个人借用记录、取消待审批的借用申请、确认归还已借用的设备。学生无法访问设备管理、用户管理、审批管理等管理员功能。

### 3.2 用户管理需求

**登录与认证**：用户通过用户名（学号或工号）和密码登录系统。系统使用 BCrypt 算法对密码进行加密存储，登录成功后签发 JWT Token，Token 有效期为 24 小时。用户登出时，Token 被加入 Redis 黑名单以防止被重复使用。支持 Token 刷新机制，在 Token 即将过期时可以无感续期。

**注册功能**：新用户可以自行注册账号，注册时需填写用户名、密码、真实姓名，并选择角色类型（学生或教师）。注册成功后默认为启用状态，管理员可以后续修改用户状态。用户名必须唯一，密码使用 BCrypt 加密后存储。

**个人信息管理**：用户登录后可以查看和修改个人基本信息，包括真实姓名、手机号、邮箱等。用户名和角色信息不可自行修改。

**用户管理（管理员）**：管理员可以分页查询所有用户列表，支持按用户名或姓名模糊搜索，支持按角色类型筛选。管理员可以启用或禁用用户账号，被禁用的用户无法登录系统。管理员可以查看实验室的成员列表。

**实验室管理（管理员）**：管理员可以新增、编辑实验室信息，包括实验室名称、位置、负责人、联系电话等。每个实验室可以关联多名用户成员。

### 3.3 设备管理需求

**设备分类管理**：系统支持树形结构的设备分类体系，管理员可以创建多级分类（如：电子仪器 → 示波器/信号发生器）。每个分类包含名称、父级分类和排序字段。支持分类的新增、编辑和删除操作，删除分类前需确认该分类下没有关联设备。

**设备台账管理**：管理员可以对设备进行完整的 CRUD 操作。每台设备包含以下核心属性：设备编号（唯一资产编号，如 DEV-2024-0001）、设备名称、规格型号、所属分类、所属实验室、总数量、当前可用数量、计量单位、单价、购置日期、当前状态（正常/维修中/已报废/外借中）、存放具体位置、备注描述。系统支持分页查询设备列表，可按名称关键词、分类、实验室、状态等多维度筛选。

**设备状态流转**：设备具有四种基本状态，状态之间的流转有明确的业务规则。正常状态的设备可以变更为维修中或外借中；维修中的设备修复完成后恢复为正常状态，若无法修复则变更为已报废状态。所有状态变更均记录操作时间和操作人。

**库存管理**：每台设备维护 `total_quantity`（总数量）和 `available_quantity`（可用数量）两个字段。总数量在设备录入时设定，可用数量随借用、归还、审批等操作动态变化。库存的扣减和回增操作必须使用乐观锁机制（基于 `version` 字段）确保并发安全。

**维修记录管理**：用户可以提交设备报修申请，填写故障描述。维修记录包含设备ID、报修人、故障描述、维修状态（待处理/维修中/已修复/无法修复）、维修费用、维修厂商、修复时间等字段。提交报修申请时，设备状态自动变更为维修中；维修完成后，设备状态恢复为正常。

### 3.4 借用预约需求

借用预约是系统最核心的业务功能，涉及复杂的状态流转和库存并发控制机制。

**提交借用申请**：学生或教师可以提交设备借用申请，需填写以下信息：借用设备（从可用设备列表中选择）、借用数量、预计借用开始时间、预计归还时间、用途说明。系统在接收到借用申请后，首先通过 Redis 分布式锁对目标设备加锁（锁键为 `lock:device:{deviceId}`，超时时间 30 秒），然后查询设备的当前可用数量和版本号，若可用数量充足，则通过乐观锁 SQL 扣减可用库存，同时创建借用申请记录（状态为待审批）。若库存不足或乐观锁更新失败（影响行数为 0），系统拒绝申请并提示用户库存不足或操作频繁。

**审批流程**：教师或管理员可以查看待审批的借用申请列表，对每条申请执行批准或驳回操作。审批通过后，借用记录状态变更为"已批准（借用中）"，由于申请阶段已经预占了库存，审批通过时无需再次操作库存。审批驳回后，借用记录状态变更为"已驳回"，系统需要释放之前预占的库存（通过乐观锁回加 `available_quantity`），并记录驳回理由通知申请人。

**归还确认**：借用人可以在借用记录列表中确认归还设备。归还时需填写归还备注（如有损坏情况需说明）。系统更新借用记录的状态为"已归还"，记录实际归还时间，同时通过乐观锁回加设备的可用库存。

**取消申请**：仅在待审批状态下的借用申请可以被取消。取消操作由申请人本人发起，系统将借用记录状态变更为"已取消"，并释放之前预占的库存。

**逾期处理**：系统通过定时任务（每小时执行一次）自动扫描所有"已批准（借用中）"状态的借用记录，若当前时间已超过预计归还时间（`end_time`），则自动将状态标记为"已逾期"。逾期记录在管理员和教师界面中以红色标签高亮显示，提醒及时跟进处理。逾期用户确认归还后，状态变更为"已归还"。

**我的借用记录**：用户可以查看个人的所有借用申请记录，支持按状态筛选（全部/待审批/已批准/已归还/已逾期/已驳回/已取消），记录按申请时间倒序排列，展示申请单号、设备名称、借用时间、应还时间、状态等关键信息。

### 3.5 统计报表需求

**首页仪表盘**：系统首页以卡片形式展示四个核心统计指标：设备总数、活跃借用数、待审批数量、逾期未还数量。这些指标通过聚合查询实时计算，为管理员提供全局概览。

**设备使用率 Top10**：通过 Echarts 柱状图展示借用次数最多的前 10 台设备，包括设备名称、总借用次数、累计借用数量和使用率百分比。使用率计算公式为：`累计借用数量 / 设备总数量 × 100%`。

**月度借用趋势图**：通过 Echarts 折线图和柱状图的组合图表展示近 12 个月的借用趋势数据，折线表示每月借用次数，柱状表示每月借用总数量，帮助管理者了解设备使用的时间分布规律。

## 4. 业务流程分析

### 4.1 设备借用业务流程

设备借用是系统最核心的业务流程，涉及多个角色和系统组件的协作。完整的借用业务流程如下：

1. **发起申请**：学生或教师在设备预约页面选择目标设备，填写借用数量、预计借用时间范围和用途说明后提交借用申请。
2. **库存预占**：系统接收申请后，首先通过 Redis 分布式锁（键名 `lock:device:{deviceId}`，超时 30 秒）对目标设备加锁，防止并发冲突。加锁成功后，查询设备的 `available_quantity` 和 `version` 字段，若可用数量充足，执行乐观锁 SQL 语句扣减可用库存。乐观锁 SQL 为：`UPDATE device_info SET available_quantity = available_quantity - #{quantity}, version = version + 1 WHERE id = #{id} AND version = #{version} AND available_quantity >= #{quantity}`。若影响行数为 1 表示扣减成功，若为 0 则表示库存不足或版本冲突（被其他并发请求修改），系统拒绝申请并释放分布式锁。
3. **创建记录**：库存扣减成功后，系统创建借用申请记录，状态设为"待审批（0）"，记录申请人ID、设备ID、借用数量、起止时间、用途等信息。
4. **教师审批**：教师或管理员在待审批列表中查看申请，可以选择"通过"或"驳回"。
   - **审批通过**：将借用记录状态更新为"已批准（1）"，记录审批人和审批时间。由于申请阶段已通过乐观锁预占了库存，审批通过时无需再次操作库存。
   - **审批驳回**：将借用记录状态更新为"已驳回（4）"，记录驳回理由。同时通过乐观锁回加设备可用库存，SQL 为：`UPDATE device_info SET available_quantity = available_quantity + #{quantity}, version = version + 1 WHERE id = #{id} AND version = #{version}`。
5. **设备使用**：审批通过后，学生按预约时间到指定地点领取设备，设备处于"借用中"状态。
6. **归还设备**：学生使用完毕后在系统中确认归还，填写归还备注（如有损坏情况说明）。系统将借用记录状态更新为"已归还（2）"，记录实际归还时间，同时通过乐观锁回加设备可用库存。
7. **逾期处理**：系统定时任务每小时扫描"已批准"状态且已超过预计归还时间的记录，自动标记为"已逾期（3）"。逾期用户归还设备后，状态变为"已归还"。
8. **取消申请**：仅"待审批"状态的申请可以被申请人取消。取消后系统释放预占库存。

### 4.2 设备维修业务流程

1. **报修提交**：用户在设备详情页面发现设备故障，点击"报修"按钮，填写故障描述后提交。
2. **状态变更**：系统创建维修记录（状态：待处理），同时将设备状态从"正常"更新为"维修中"，该设备在此期间不可被借用预约。
3. **维修处理**：维修人员或管理员查看待处理维修列表，将维修状态更新为"维修中"，可补充维修厂商信息。
4. **维修完成**：维修完成后，将维修状态更新为"已修复"，记录维修费用和修复时间，设备状态恢复为"正常"，重新可被借用。
5. **无法修复**：若设备无法修复，将维修状态更新为"无法修复"，设备状态变更为"已报废"。

### 4.3 逾期处理流程

1. **定时扫描**：系统配置 Spring `@Scheduled` 定时任务，Cron 表达式为 `0 0 * * * ?`（每小时整点执行）。
2. **状态标记**：查询所有 `status = 1`（已批准/借用中）且 `end_time < NOW()` 的借用记录，批量更新为 `status = 3`（已逾期）。
3. **通知提醒**：逾期记录在管理界面中以红色标签高亮显示，管理员和教师可以查看逾期列表并跟进处理。
4. **归还处理**：逾期用户归还设备后，记录实际归还时间和归还备注，状态变更为"已归还"。

## 5. 非功能需求

### 5.1 性能需求

- 页面首次加载时间不超过 2 秒（在校园网环境下）。
- 单个 API 接口的响应时间不超过 500 毫秒（正常负载下）。
- 系统应支持至少 100 个用户同时在线操作，借用申请接口（`/borrow/apply`）需通过乐观锁和分布式锁保证并发安全性，在 10 个用户同时借用仅剩 1 件的设备时，最终只有 1 人成功。
- 设备列表分页查询支持至少 1000 条设备记录的快速检索，响应时间不超过 1 秒。
- Redis 操作响应时间不超过 10 毫秒。

### 5.2 安全需求

- 所有 API 接口必须通过 JWT Token 进行身份认证，Token 通过 HTTP Header `Authorization` 传递。
- 用户密码使用 BCrypt 算法加密存储，禁止明文存储和传输。
- API 网关负责统一的 JWT 解析和用户信息传递，下游微服务通过请求头（`X-User-Id`、`X-User-Role`）获取当前用户信息。
- 系统实现基于角色的访问控制（RBAC），前端路由守卫和后端接口均进行权限校验，学生角色无法访问管理员专属接口（如设备新增、用户管理等），越权访问返回 403 状态码。
- 防止 SQL 注入：使用 MyBatis-Plus 的参数化查询，禁止拼接 SQL 字符串。
- 防止 XSS 攻击：前端对用户输入进行转义处理。
- JWT 登出机制：登出时 Token 加入 Redis 黑名单，防止 Token 被复用。

### 5.3 可靠性需求

- 数据库所有写入操作使用 `@Transactional` 事务注解，确保数据一致性。库存扣减和借用记录创建在同一事务中执行，任何步骤失败均回滚。
- 设备库存的并发操作通过乐观锁（`version` 字段）保证数据一致性，避免超借现象。乐观锁更新失败时系统抛出明确的异常信息，提示用户重试。
- 高并发场景下通过 Redis 分布式锁（键名 `lock:device:{deviceId}`）避免大量乐观锁重试导致的数据库压力。
- 系统提供全局异常处理器（`@RestControllerAdvice`），统一捕获和处理各类异常，返回标准格式的错误响应，避免暴露系统内部信息。
- Feign 服务间调用配置 FallbackFactory 降级策略，当下游服务不可用时返回友好的降级提示，避免级联故障。

### 5.4 可用性需求

- 系统采用响应式页面设计，适配主流桌面浏览器（Chrome 80+、Firefox 80+、Edge 80+）。
- 页面布局清晰，操作流程直观，重要操作（如删除设备、驳回申请）需二次确认。
- 所有 API 错误响应使用统一的 `R<T>` 格式，包含错误码和可读的错误消息。
- 表单输入提供即时校验反馈（如必填项校验、格式校验）。
- 列表数据支持分页展示，避免一次性加载大量数据。

### 5.5 兼容性需求

- 后端兼容 JDK 1.8 运行环境，不依赖 JDK 11+ 特性。
- 前端兼容现代浏览器，不兼容 IE 浏览器。
- 数据库兼容 MySQL 8.0 系列，字符集使用 `utf8mb4`。
- 部署环境支持 Windows Server 和 Linux（CentOS/Ubuntu）。
- 中间件版本锁定：Nacos 2.2.3、Redis 5.0+。

## 6. 约束条件

### 6.1 技术约束

- 后端开发语言：Java 8（JDK 1.8）
- 后端框架：Spring Boot 2.7.18、Spring Cloud 2021.0.8、Spring Cloud Alibaba 2021.0.5.0
- 持久层框架：MyBatis-Plus 3.5.5
- 前端框架：Vue 3.3.13 + Vite 4.5.3 + Element Plus 2.7.6
- 数据库：MySQL 8.0.36，字符集 utf8mb4
- 服务注册与配置中心：Nacos 2.2.3
- 缓存：Redis 5.0+
- Web 服务器：Nginx 1.24.0

### 6.2 开发约束

- 前后端必须完全分离，通过 RESTful API 进行数据交互。
- 所有 API 接口遵循统一的响应格式 `R<T>`（包含 code、message、data、timestamp 字段）。
- 分页查询返回格式：`{ records: [], total: 100, size: 10, current: 1 }`。
- 所有依赖版本必须在 `pom.xml` 和 `package.json` 中硬编码，禁止使用动态版本。
- 禁止在 API 响应中返回 `password_hash` 等敏感字段。

### 6.3 部署约束

- 目标部署环境为 Windows Server（同时提供 Linux 部署支持）。
- 提供 Windows 一键启动脚本 `start-lab.bat`。
- 前端构建产物（`dist/` 目录）通过 Nginx 进行静态资源托管和反向代理。
- 所有微服务通过 Nacos 进行服务注册和发现，网关统一对外暴露 API。
