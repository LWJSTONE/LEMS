## 1. 架构概述

### 1.1 设计原则

LEMS 系统采用前后端分离的微服务架构设计，遵循以下核心设计原则：

- **单一职责原则**：每个微服务只负责一个明确的业务领域，服务之间通过定义良好的 API 进行通信，避免服务间的功能交叉和耦合。例如，设备管理服务专注于设备 CRUD 和库存操作，借用服务专注于借用流程管理，两者通过 OpenFeign 进行跨服务调用。
- **高内聚低耦合**：将业务紧密相关的功能聚合在同一个微服务内（如设备分类、设备台账、维修记录都属于设备管理服务的范畴），服务之间通过 RESTful API 或消息队列进行松耦合通信。
- **前后端分离**：前端（Vue 3 SPA）和后端（Spring Cloud 微服务）完全独立开发、独立部署，通过标准的 RESTful JSON API 进行数据交互。前端通过 Nginx 托管静态资源，所有 API 请求通过 Nginx 反向代理转发到 API 网关。
- **面向失败设计**：Feign 调用配置 FallbackFactory 降级策略，当下游服务不可用时返回默认值而非抛出异常；Redis 分布式锁设置超时时间防止死锁；事务边界清晰定义，避免长事务。
- **版本锁定**：所有依赖版本在 `pom.xml` 和 `package.json` 中硬编码，禁止使用动态版本范围（如 `2.7.x`），确保构建的可重复性。

### 1.2 架构风格

系统采用**基于 API 网关的微服务架构**，这是当前 Spring Cloud 生态中最主流的微服务架构模式。所有外部请求首先经过 API 网关，由网关负责请求路由、JWT 鉴权、跨域处理等横切关注点，下游微服务无需重复实现这些功能。微服务之间通过 OpenFeign 进行同步 HTTP 调用，使用 Nacos 作为服务注册与配置中心。

## 3. 微服务拆分设计

### 3.1 服务划分及职责

系统按照业务领域划分为以下 7 个模块，其中 1 个公共模块和 6 个可独立部署的微服务：

| 模块名称 | 端口 | 类型 | 核心职责 |
|---------|------|------|---------|
| lab-common | - | 公共库 | 父 POM 依赖管理、统一返回体 `R<T>`、全局异常处理器、基础实体 `BaseEntity`、JWT 工具类、Feign 接口定义及降级工厂 |
| lab-gateway | 8080 | 微服务 | API 网关，负责请求路由分发、JWT Token 解析与验证、用户信息传递（X-User-Id/X-User-Role）、跨域配置、白名单放行（登录/注册接口） |
| lab-auth | 8081 | 微服务 | 用户认证服务，负责登录注册、密码 BCrypt 加密、JWT Token 签发与刷新、登出黑名单管理 |
| lab-user | 8082 | 微服务 | 用户管理服务，负责用户档案 CRUD、实验室信息管理、实验室成员管理、账号状态管理 |
| lab-device | 8083 | 微服务 | 设备管理服务，负责设备分类树 CRUD、设备台账 CRUD、库存管理（乐观锁并发控制）、维修记录管理、设备状态流转 |
| lab-borrow | 8084 | 微服务 | 借用预约服务，负责借用申请（含库存预占）、审批流程、归还确认、逾期扫描（定时任务）、取消申请 |
| lab-report | 8085 | 微服务 | 统计报表服务，负责首页仪表盘统计、设备使用率 Top10、月度借用趋势分析 |

### 3.2 服务拆分理由

- **lab-common** 作为公共模块不独立部署，以 JAR 依赖的形式被其他微服务引用。将所有服务通用的代码（统一返回体、异常处理、工具类、Feign 接口定义）集中管理，避免代码重复，保证各服务对通用功能的实现一致性。
- **lab-gateway** 作为系统唯一入口，将鉴权、路由、跨域等横切关注点从业务服务中剥离，使业务服务只需关注自身业务逻辑，降低了系统的复杂度。网关通过解析 JWT Token 并将用户信息放入 HTTP Header 传递给下游服务，下游服务无需直接解析 JWT。
- **lab-auth** 独立为认证服务是因为认证逻辑（密码校验、Token 签发）具有独立性和通用性，且涉及安全敏感操作（BCrypt 加密、Redis 黑名单），适合独立部署和扩展。
- **lab-user** 独立为用户服务是因为用户数据（账号、角色、所属实验室）是多个业务服务的依赖基础，独立出来便于统一管理和被其他服务通过 Feign 调用。
- **lab-device** 独立为设备服务是因为设备管理（台账 CRUD、分类管理、维修记录）是系统的核心数据基础，且涉及关键的库存并发控制逻辑（乐观锁），需要独立管理事务和数据一致性。
- **lab-borrow** 独立为借用服务是因为借用流程是最复杂的业务逻辑，涉及状态机（待审批→已批准→已归还/已逾期/已驳回/已取消）、库存预占与释放、跨服务调用（通过 Feign 调用设备服务的库存操作和用户服务的信息查询）、定时任务（逾期扫描），适合独立为一个服务。
- **lab-report** 独立为报表服务是因为统计查询通常是读密集型操作，可能涉及复杂的聚合 SQL，独立出来便于后续单独优化查询性能或引入缓存策略，不影响核心业务服务的性能。

### 3.3 服务间通信

服务间通过 OpenFeign 进行同步 HTTP 调用，Feign 接口统一定义在 `lab-common` 模块的 `feign` 包下：

```
lab-common/feign/
├── UserFeignClient.java          # 用户服务客户端
│   ├── getUserById(id)           # 获取用户信息
│   └── getUserPhone(id)          # 获取用户手机号
├── UserFeignClientFallbackFactory.java  # 用户服务降级工厂
├── DeviceFeignClient.java        # 设备服务客户端
│   ├── getDeviceInfo(id)         # 获取设备详情（含版本号）
│   └── updateAvailableQuantity(params)  # 更新可用库存
└── DeviceFeignClientFallbackFactory.java  # 设备服务降级工厂
```

**主要调用链路**：

- `lab-borrow` → `lab-device`：借用申请时查询设备详情和扣减库存，归还/取消/驳回时回加库存。
- `lab-borrow` → `lab-user`：借用申请后获取借用人联系方式（可选扩展）。

所有 Feign 接口均配置了 FallbackFactory 降级工厂，当下游服务不可用时返回友好的错误提示，避免级联故障导致系统整体不可用。

## 4. 网关设计

### 4.1 路由规则

API 网关（lab-gateway）基于 Spring Cloud Gateway 实现，采用服务发现模式（通过 Nacos 获取服务实例列表）进行负载均衡路由。路由规则如下：

| 路由ID | 匹配路径 | 目标服务 | 说明 |
|-------|---------|---------|------|
| lab-auth | `/api/v1/auth/**` | lb://lab-auth | 认证服务 |
| lab-user | `/api/v1/user/**` | lb://lab-user | 用户服务 |
| lab-device | `/api/v1/device/**` | lb://lab-device | 设备服务 |
| lab-borrow | `/api/v1/borrow/**` | lb://lab-borrow | 借用服务 |
| lab-report | `/api/v1/report/**` | lb://lab-report | 报表服务 |

### 4.2 JWT 鉴权过滤器

网关的核心组件是 `JwtAuthGlobalFilter`（实现 `GlobalFilter` 和 `Ordered` 接口），其工作流程如下：

1. **白名单判断**：对于 `/api/v1/auth/login` 和 `/api/v1/auth/register` 路径直接放行，不进行 Token 校验。
2. **Token 提取**：从 HTTP Header `Authorization` 中提取 Token，校验格式是否以 `Bearer ` 为前缀。
3. **黑名单检查**：查询 Redis 中是否存在 `jwt:blacklist:{token}` 键，若存在则 Token 已失效（用户已登出），返回 401。
4. **Token 解析**：使用 JWT 工具类解析 Token，获取 `userId`、`username`、`role` 等信息。
5. **过期检查**：验证 Token 是否已过期，过期则返回 401。
6. **信息传递**：将解析出的用户信息通过 HTTP Header 传递给下游微服务：`X-User-Id`（用户ID）、`X-Username`（用户名）、`X-User-Role`（角色）。

### 4.3 跨域配置

网关全局配置 CORS，允许所有来源、所有方法、所有请求头的跨域访问，支持携带凭证（Credentials）。这确保前端开发环境（Vite dev server 在 :3000 端口）可以正常调用后端 API。

### 5.2 核心关系说明

1. **sys_user ↔ lab_info**：多对一关系。多个用户可以属于同一个实验室（`sys_user.lab_id → lab_info.id`），一个实验室有一个负责人（`lab_info.manager_id → sys_user.id`）。
2. **device_info ↔ device_category**：多对一关系。多台设备属于同一个分类（`device_info.category_id → device_category.id`）。
3. **device_info ↔ lab_info**：多对一关系。多台设备属于同一个实验室（`device_info.lab_id → lab_info.id`）。
4. **device_info ↔ borrow_record**：一对多关系。一台设备可以被多次借用（`borrow_record.device_id → device_info.id`）。
5. **sys_user ↔ borrow_record**：一对多关系。一个用户可以有多条借用记录（`borrow_record.user_id → sys_user.id`），同时一个用户也可以作为审批人出现（`borrow_record.approver_id → sys_user.id`）。
6. **device_info ↔ device_maintenance**：一对多关系。一台设备可以有多条维修记录。
7. **device_category** 自引用：通过 `parent_id` 实现树形分类结构。
8. **sys_dict**：复合主键表，`dict_type + dict_code` 联合唯一，用于存储系统级的数据字典（设备状态、借用状态、角色类型、维修状态等）。

### 6.2 库存并发控制架构

库存并发控制是系统最核心的技术难点，采用 **Redis 分布式锁 + 数据库乐观锁** 双重保障机制：

```
用户提交借用申请
       │
       ▼
 ┌─────────────────┐
 │ 1. 获取Redis    │
 │    分布式锁     │──失败──→ 提示"操作频繁，请稍后重试"
 │ lock:device:ID  │
 └────────┬────────┘
          │ 成功
          ▼
 ┌─────────────────┐
 │ 2. 查询设备     │
 │    详情(含version)│
 └────────┬────────┘
          │
          ▼
 ┌─────────────────────────────┐
 │ 3. 校验库存                   │
 │    available >= borrow_qty?  │──否──→ 提示"库存不足"
 └────────┬────────────────────┘
          │ 是
          ▼
 ┌──────────────────────────────────────────┐
 │ 4. 乐观锁扣减库存 (在同一个事务中)         │
 │                                            │
 │ UPDATE device_info                         │
 │ SET available_quantity = available_quantity│
 │     - #{quantity},                         │
 │     version = version + 1                  │
 │ WHERE id = #{id}                           │
 │   AND version = #{version}                 │
 │   AND available_quantity >= #{quantity}    │
 │                                            │
 │ 影响行数 = 1 → 成功                         │
 │ 影响行数 = 0 → 库存不足或版本冲突           │
 └────────┬──────────────────────────────────┘
          │ 成功
          ▼
 ┌─────────────────┐
 │ 5. 创建借用记录  │
 │    状态: 待审批  │
 └────────┬────────┘
          │
          ▼
 ┌─────────────────┐
 │ 6. 释放Redis锁   │
 └─────────────────┘
```

```
│       └── lab/            # 实验室管理

│   └── Layout.vue          # 布局组件（侧边栏+顶部栏+内容区）

```

### 7.3 路由权限守卫

前端路由通过 Vue Router 的 `beforeEach` 全局前置守卫实现基于角色的菜单权限控制：

- **未登录**：所有页面重定向到 `/login`。
- **已登录**：根据路由元信息 `meta.roles` 判断当前用户角色是否有权访问。例如 `meta.roles: ['ADMIN']` 表示该路由仅管理员可见，学生和教师角色访问时自动跳转到首页仪表盘。
- **角色对应菜单**：管理员可以看到所有菜单（设备分类管理、用户管理、实验室管理、借用审批等），教师可以看到借用审批菜单但看不到用户管理菜单，学生只能看到设备列表、设备预约和我的借用菜单。

## 8. 安全架构

### 8.1 认证流程

```
用户登录请求 → lab-auth (POST /api/v1/auth/login)
    │
    ├─ 验证用户名密码 (BCrypt 校验)
    ├─ 检查账号状态 (是否启用)
    ├─ 签发 JWT Token (有效期 24h)
    │   Payload: { userId, username, role }
    │   签名算法: HS512
    └─ 返回 Token + 用户信息

后续请求 → Nginx → Gateway
    │
    ├─ 提取 Authorization Header
    ├─ 检查 Redis 黑名单
    ├─ 解析 JWT → 提取 userId, username, role
    └─ 注入 X-User-Id, X-User-Role Header → 下游服务

用户登出 → Token 加入 Redis 黑名单 (TTL = Token剩余有效期)
```

### 8.2 权限控制层次

| 层次 | 实现方式 | 说明 |
|------|---------|------|
| 网关层 | JwtAuthGlobalFilter | 校验 Token 有效性和黑名单，未携带 Token 返回 401 |
| 前端路由层 | router.beforeEach | 根据 `meta.roles` 隐藏无权限菜单，未登录重定向到登录页 |
| 前端UI层 | v-if="isAdmin" | 根据角色控制按钮的显示和隐藏（如"新增设备"按钮仅管理员可见） |
| 后端接口层 | @RequestHeader | 从网关传递的 Header 获取用户身份信息，可进一步校验角色权限 |
