## 1. 数据库概述

### 1.1 设计规范

本系统数据库设计遵循以下规范，以确保数据的一致性、完整性和查询性能：

- **存储引擎**：统一使用 InnoDB 引擎，支持事务（ACID）、行级锁和外键约束。
- **字符集**：使用 `utf8mb4` 字符集，支持完整的 Unicode 字符（包括 Emoji），排序规则为 `utf8mb4_general_ci`。
- **主键策略**：使用 MyBatis-Plus 的 `ASSIGN_ID` 策略（雪花算法 Snowflake）生成 64 位 BIGINT 类型主键，保证全局唯一且趋势递增，有利于 B+ 树索引的性能。
- **逻辑删除**：大部分业务表包含 `deleted` 字段（TINYINT，0 表示正常、1 表示已删除），使用 MyBatis-Plus 的 `@TableLogic` 注解自动处理逻辑删除，查询时自动过滤已删除数据，删除时执行 UPDATE 而非 DELETE。
- **审计字段**：所有业务表包含 `create_time`（创建时间）和 `update_time`（更新时间）字段，`create_time` 在插入时自动填充，`update_time` 在插入和更新时自动填充。
- **乐观锁**：涉及并发修改的表（`device_info`、`borrow_record`）包含 `version` 字段（INT），使用 MyBatis-Plus 的 `@Version` 注解自动管理乐观锁版本号。
- **命名约定**：表名使用小写下划线命名法（如 `borrow_record`），字段名使用小写下划线命名法（如 `available_quantity`），与 Java 实体类的驼峰命名通过 MyBatis-Plus 自动映射。

### 1.2 ER 关系总结

| 关系 | 类型 | 说明 |
|------|------|------|
| sys_user → lab_info | N:1 | 多用户属于一实验室 |
| lab_info → sys_user | N:1 | 实验室有一个负责人 |
| device_info → device_category | N:1 | 多设备属于一分类 |
| device_info → lab_info | N:1 | 多设备属于一实验室 |
| device_info → borrow_record | 1:N | 一设备有多条借用记录 |
| sys_user → borrow_record | 1:N | 一用户有多条借用记录 |
| sys_user → borrow_record (approver) | 1:N | 一用户可审批多条记录 |
| device_info → device_maintenance | 1:N | 一设备有多条维修记录 |
| device_category → device_category (parent) | 1:N | 树形自引用分类结构 |

存储系统所有用户的基本信息和认证数据。

|--------|------|------|------|

| username | VARCHAR(50) | UK, NOT NULL | 学号/工号，登录账号 |
| password_hash | VARCHAR(100) | NOT NULL | BCrypt 加密后的密码密文 |

| phone | VARCHAR(20) | NULL | 手机号码 |
| email | VARCHAR(50) | NULL | 电子邮箱 |
| role_type | VARCHAR(20) | NOT NULL | 角色类型：ADMIN / TEACHER / STUDENT |
| lab_id | BIGINT | NULL | 所属实验室 ID（关联 lab_info.id） |
| status | TINYINT | NOT NULL, DEFAULT 1 | 账号状态：0=禁用，1=启用 |
| deleted | TINYINT | NOT NULL, DEFAULT 0 | 逻辑删除：0=正常，1=已删除 |
| create_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP | 创建时间 |
| update_time | DATETIME | NOT NULL, DEFAULT CURRENT_TIMESTAMP ON UPDATE | 更新时间 |

**索引设计**：

| 索引名 | 类型 | 字段 | 说明 |
|--------|------|------|------|
| PRIMARY | 主键 | id | 聚簇索引 |
| uk_username | 唯一索引 | username | 保证用户名全局唯一 |
| idx_role_type | 普通索引 | role_type | 加速按角色筛选查询 |
| idx_lab_id | 普通索引 | lab_id | 加速查询实验室成员 |

**设计说明**：
- `username` 设置唯一索引，确保学号/工号不会重复。
- `password_hash` 使用 BCrypt 算法（强度因子 10），生成的密文长度为 60 字符，VARCHAR(100) 预留足够空间。
- `role_type` 使用 VARCHAR 而非 TINYINT，便于代码中的枚举映射，且可读性更好。
- `lab_id` 允许 NULL，因为管理员可能不属于任何具体实验室。

| name | VARCHAR(50) | NOT NULL | 分类名称 |

| sort_order | INT | NOT NULL, DEFAULT 0 | 同级排序号 |
| deleted | TINYINT | NOT NULL, DEFAULT 0 | 逻辑删除 |
| create_time | DATETIME | NOT NULL | 创建时间 |

**索引设计**：

| 索引名 | 类型 | 字段 | 说明 |
|--------|------|------|------|

### 2.4 device_info（设备台账表）⭐ 核心表

存储所有设备的基本信息、库存数据和状态信息，是系统最核心的业务表。

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PK | 设备 ID |
| code | VARCHAR(50) | UK, NOT NULL | 设备唯一编号/资产号 |
| name | VARCHAR(100) | NOT NULL | 设备名称 |
| model | VARCHAR(100) | NULL | 规格型号 |
| category_id | BIGINT | NULL | 分类 ID（关联 device_category.id） |
| lab_id | BIGINT | NULL | 所属实验室 ID（关联 lab_info.id） |
| total_quantity | INT | NOT NULL, DEFAULT 1 | 设备总数量 |
| available_quantity | INT | NOT NULL, DEFAULT 1 | 当前可用数量（库存并发控制核心字段） |
| unit | VARCHAR(20) | NULL, DEFAULT '台' | 计量单位 |
| price | DECIMAL(12,2) | NULL | 单价（元） |
| purchase_date | DATE | NULL | 购置日期 |
| status | TINYINT | NOT NULL, DEFAULT 0 | 设备状态：0=正常，1=维修中，2=报废，3=外借 |
| location_detail | VARCHAR(255) | NULL | 存放具体位置（如：A301-柜1） |
| description | TEXT | NULL | 备注说明 |
| version | INT | NOT NULL, DEFAULT 0 | 乐观锁版本号（库存并发控制核心字段） |
| deleted | TINYINT | NOT NULL, DEFAULT 0 | 逻辑删除 |
| create_time | DATETIME | NOT NULL | 创建时间 |
| update_time | DATETIME | NOT NULL | 更新时间 |

**索引设计**：

| 索引名 | 类型 | 字段 | 说明 |
|--------|------|------|------|
| PRIMARY | 主键 | id | 聚簇索引 |
| uk_code | 唯一索引 | code | 保证设备编号全局唯一 |
| idx_lab_category | 复合索引 | (lab_id, category_id) | 加速按实验室+分类组合查询 |
| idx_status | 普通索引 | status | 加速按状态筛选（如查询正常设备） |
| idx_name | 普通索引 | name | 加速按名称模糊搜索 |

**设计说明**：
- **`available_quantity` 是库存并发控制的核心字段**。借用申请时乐观锁扣减，归还/取消/驳回时乐观锁回加。该字段必须满足 `available_quantity >= 0` 的约束，通过乐观锁 SQL 的 WHERE 条件 `available_quantity >= #{quantity}` 保证。
- **`version` 是乐观锁版本号**，每次更新（包括库存扣减和回加）时自动递增。MyBatis-Plus 的 `@Version` 注解在 UPDATE 语句中自动添加 `AND version = #{version}` 条件和 `SET version = version + 1` 赋值。
- **乐观锁扣减 SQL**：`UPDATE device_info SET available_quantity = available_quantity - #{quantity}, version = version + 1 WHERE id = #{id} AND version = #{version} AND available_quantity >= #{quantity}`。影响行数为 1 表示成功，为 0 表示库存不足或版本冲突。
- `code` 字段设置唯一索引，作为设备的唯一标识（资产编号），在入库时手动录入。
- `price` 使用 DECIMAL(12,2) 类型，精度为 12 位数字其中 2 位小数，最大可表示 9999999999.99 元，满足设备单价存储需求。
- `description` 使用 TEXT 类型，支持较长的备注内容。

| borrow_quantity | INT | NOT NULL, DEFAULT 1 | 借用数量 |
| start_time | DATETIME | NOT NULL | 预计借用开始时间 |
| end_time | DATETIME | NOT NULL | 预计归还时间 |
| actual_return_time | DATETIME | NULL | 实际归还时间 |
| status | TINYINT | NOT NULL, DEFAULT 0 | 状态：0=待审批，1=已批准，2=已归还，3=已逾期，4=已驳回，5=已取消 |
| approver_id | BIGINT | NULL | 审批人 ID（关联 sys_user.id） |
| approve_time | DATETIME | NULL | 审批时间 |
| reject_reason | VARCHAR(255) | NULL | 驳回理由 |
| return_remark | VARCHAR(255) | NULL | 归还备注（损坏情况说明） |
| version | INT | NOT NULL, DEFAULT 0 | 乐观锁版本号 |

**索引设计**：

| 索引名 | 类型 | 字段 | 说明 |
|--------|------|------|------|
| PRIMARY | 主键 | id | 聚簇索引 |
| idx_user_status | 复合索引 | (user_id, status) | 加速查询某用户的特定状态记录（如我的待审批记录） |
| idx_device_status | 复合索引 | (device_id, status) | 加速查询某设备的特定状态记录（如某设备的借用中记录） |
| idx_end_time | 普通索引 | end_time | 加速逾期扫描查询（`status=1 AND end_time < NOW()`） |

**设计说明**：
- `status` 字段使用 6 个状态值表示借用记录的完整生命周期。状态流转规则参见架构设计文档中的状态机图。
- `version` 字段为借用记录自身的乐观锁，用于防止对同一条借用记录的并发修改（如两个人同时对同一条记录执行审批操作）。
- `idx_end_time` 索引是专门为逾期扫描定时任务设计的，该任务每小时执行一次，查询条件为 `status = 1 AND end_time < NOW()`，该索引可以显著加速此查询。
- `actual_return_time` 仅在归还时填写，用于记录实际归还时间，便于计算逾期天数。
- `borrow_quantity` 允许一次借用多件同类设备（如借用 3 台笔记本电脑）。

## 3. 索引设计依据

### 3.1 索引类型选择

| 索引类型 | 适用场景 | 本系统中的使用 |
|---------|---------|--------------|
| 主键索引 (PRIMARY) | 所有表的 ID 字段 | 所有表均使用 |
| 唯一索引 (UNIQUE) | 需要保证字段值全局唯一 | sys_user.username, device_info.code |
| 普通索引 (INDEX) | 高频查询条件的等值或范围查询 | status, role_type, parent_id, end_time |
| 复合索引 (INDEX) | 多字段组合查询条件 | (user_id, status), (device_id, status), (lab_id, category_id) |

### 3.2 核心查询与索引匹配

| 查询场景 | SQL 条件 | 使用的索引 | 说明 |
|---------|---------|-----------|------|
| 用户登录 | WHERE username = ? | uk_username | 唯一索引，O(1) 查找 |
| 设备列表分页 | WHERE name LIKE ? AND status = ? | idx_name + idx_status | 组合使用 |
| 我的借用记录 | WHERE user_id = ? AND status = ? | idx_user_status | 复合索引，覆盖索引 |
| 待审批列表 | WHERE status = 0 | idx_device_status | 借用记录表 |
| 逾期扫描 | WHERE status = 1 AND end_time < NOW() | idx_end_time | 定时任务专用索引 |
| 实验室设备 | WHERE lab_id = ? AND category_id = ? | idx_lab_category | 复合索引 |
| 设备库存扣减 | WHERE id = ? AND version = ? AND available_quantity >= ? | PRIMARY (id) | 乐观锁使用主键查找 |

## 4. 乐观锁并发控制设计

### 4.1 device_info 表的库存控制

`device_info` 表的 `version` 和 `available_quantity` 字段是实现库存并发控制的核心：

- **扣减库存**（借用申请时）：
  ```sql
  UPDATE device_info
  SET available_quantity = available_quantity - #{quantity},
      version = version + 1,
      update_time = NOW()
  WHERE id = #{id}
    AND version = #{version}
    AND available_quantity >= #{quantity}
  ```
  - `version = #{version}`：确保从查询到更新之间没有被其他事务修改。
  - `available_quantity >= #{quantity}`：双重保障，确保不会扣成负数。
  - 影响行数 = 1：成功；影响行数 = 0：库存不足或版本冲突，事务回滚。

- **回加库存**（归还/取消/驳回时）：
  ```sql
  UPDATE device_info
  SET available_quantity = available_quantity + #{quantity},
      version = version + 1,
      update_time = NOW()
  WHERE id = #{id}
    AND version = #{version}
  ```
  - 不需要 `available_quantity` 的条件判断（库存只会增加）。
  - 影响行数 = 1：成功；影响行数 = 0：版本冲突。

### 4.2 borrow_record 表的状态控制

`borrow_record` 表的 `version` 字段用于防止对同一条借用记录的并发修改：

- 在审批操作中，教师 A 和教师 B 同时审批同一条记录时，只有先执行的操作会成功（version 匹配），后执行的操作会因 version 不匹配而失败，系统提示"该申请已审批"。

### 4.3 Redis 分布式锁增强

针对高并发的热门设备（如笔记本电脑），在数据库乐观锁的基础上增加了 Redis 分布式锁：

- **锁键**：`lock:device:{deviceId}`
- **锁超时**：30 秒（防止死锁）
- **加锁方式**：`SET lock:device:{id} 1 NX EX 30`（原子操作）
- **释放方式**：借用申请完成后在 `finally` 块中执行 `DEL lock:device:{id}`

分布式锁的作用是避免大量并发请求同时到达数据库执行乐观锁查询和更新，减少数据库的 CPU 和 IO 压力。即使 Redis 分布式锁获取失败，系统也能通过数据库乐观锁保证数据一致性，分布式锁仅作为性能优化手段。
